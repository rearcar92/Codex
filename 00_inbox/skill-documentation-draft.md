## 목적

Codex에서 반복적으로 쓰는 작업 방식, 프롬프트, 절차, 참고자료를 스킬 형태로 정리하기 위한 초안입니다.

아직 아이디어가 명확하지 않으므로, 지금은 완성된 문서가 아니라 생각을 모으는 작업 공간으로 사용합니다.

## 지금 떠오르는 문제

- 반복해서 비슷한 설명을 Codex에게 다시 하고 있다.
- 작업마다 원하는 결과물의 톤, 구조, 완료 기준이 달라진다.
- 좋은 작업 방식이 생겨도 다음 작업에 재사용하기 어렵다.
- PG/정산, 포트폴리오, 데이터 분석, AI 서비스, 자동화 작업이 서로 다른 기준으로 흩어진다.

## 스킬로 만들 수 있을 것 같은 후보

### 1. PG/정산 문서화 스킬

- 결제/정산/대사 관련 내용을 구조화한다.
- 정책, 예외, 운영 리스크, 화면 영향, 데이터 기준을 함께 정리한다.
- 포트폴리오나 실무 문서로 재활용 가능하게 만든다.

### 2. 포트폴리오 문장화 스킬

- 실무 경험을 면접관이 이해하기 쉬운 문장으로 바꾼다.
- 단순 업무 나열이 아니라 문제, 판단, 실행, 결과 중심으로 정리한다.
- 서비스기획자에서 PM/PO로 확장되는 관점을 반영한다.

### 3. 데이터 분석 기획 스킬

- 분석하고 싶은 질문을 지표, 데이터 기준, 해석 관점으로 바꾼다.
- SQL이나 대시보드 작업 전에 목적과 계산 기준을 먼저 정리한다.
- 결제/정산 도메인 지표를 체계화한다.

### 4. AI 서비스 기획 스킬

- 아이디어를 사용자 문제, MVP, 핵심 플로우, 검증 계획으로 정리한다.
- 기능 욕심보다 출시 가능한 최소 단위를 먼저 잡는다.
- 개인 서비스 출시를 위한 실행 문서로 만든다.

### 5. 업무 자동화 설계 스킬

- 반복 업무를 자동화 후보로 분해한다.
- 입력, 처리, 출력, 예외, 검증 기준을 정리한다.
- 자동화 전후의 시간 절감 효과를 남긴다.

## 고민할 질문

- 나는 어떤 작업을 가장 자주 반복하는가?
- Codex에게 매번 다시 설명하는 기준은 무엇인가?
- 스킬로 만들었을 때 가장 시간을 많이 줄여줄 영역은 무엇인가?
- 단순 문서 템플릿이면 충분한가, 아니면 절차형 스킬이 필요한가?
- Obsidian 노트로만 둘 것인가, 실제 Codex skill 파일로 만들 것인가?
